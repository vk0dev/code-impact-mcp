**Легковесный pre-commit шлюз безопасности для AI-агентов.** Отвечает на вопрос «безопасно ли это изменение?» вердиктом PASS/WARN/BLOCK за секунды — без базы данных, без сложной настройки.

[English](./README.md) | [日本語](./README.ja.md) | [中文](./README.zh-CN.md) | [Español](./README.es.md)

## Зачем / Когда использовать

Используйте этот MCP-сервер, когда:

- Пользователь спрашивает: **«Что сломается, если я изменю этот файл?»**
- Пользователь спрашивает: **«Безопасно ли закоммитить этот рефакторинг?»**
- Пользователь спрашивает: **«Проверь радиус поражения перед пушем»**
- Пользователь спрашивает: **«Что зависит от этого модуля?»**
- Агенту нужен **быстрый pre-commit шлюз** перед изменением нескольких файлов
- Агенту нужен **числовой риск-скор (0-1)** для предлагаемого изменения
- Нужен анализ зависимостей **без настройки базы данных или tree-sitter**

CodeImpact MCP строит легковесный граф зависимостей с помощью ts-morph и даёт ограниченный ответ: PASS, WARN или BLOCK. Без облака, без API-ключей, полностью локально.

## Инструменты

### `gate_check`

Pre-commit шлюз безопасности. Анализирует указанные изменения и возвращает **вердикт PASS/WARN/BLOCK** с обоснованием. Используйте как ограниченный помощник принятия решений перед коммитом многофайловых изменений. BLOCK означает, что риск превышает порог. WARN — рекомендуется ревью человеком. PASS — низкий риск по данным графа.

### `analyze_impact`

Анализ радиуса поражения при изменении указанных файлов. Возвращает прямо и транзитивно затронутые файлы с риск-скором (0-1). Используйте ПЕРЕД коммитом многофайловых изменений, чтобы понять, что может сломаться. НЕ модифицирует файлы.

### `get_dependencies`

Получение связей import и importedBy для конкретного файла. Показывает, от чего зависит этот файл и что зависит от него. Используйте для оценки связанности перед рефакторингом.

### `refresh_graph`

Пересборка графа зависимостей с нуля. Вызывайте после значительного добавления/удаления файлов или если результаты кажутся устаревшими. Возвращает статистику графа: количество файлов, рёбер, время сборки и обнаруженные циклические зависимости.

## Пример диалога

**Пользователь:** «Хочу отрефакторить `src/routes.ts` — это безопасно?»

**Агент вызывает** `gate_check`:
```json
{
  "projectRoot": "/Users/you/projects/my-app",
  "files": ["src/routes.ts"],
  "threshold": 0.5
}
```

**Результат:**
```json
{
  "verdict": "WARN",
  "recommendation": "Proceed only with targeted review of affected files.",
  "riskScore": 0.35,
  "reasons": ["Risk score 0.35 is approaching threshold. Review affected files."],
  "affectedFiles": 8,
  "circularDependencies": 0
}
```

**Агент:** «Проверка шлюза вернула WARN — 8 файлов зависят от routes.ts. Проверю затронутые файлы перед внесением изменений.»

## Как это работает

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│  Agent asks  │────▶│  ts-morph     │────▶│  In-memory    │
│  "safe to    │     │  parses       │     │  dependency   │
│   change?"   │     │  imports      │     │  graph        │
└─────────────┘     └──────────────┘     └──────┬───────┘
                                                 │
                    ┌──────────────┐     ┌───────▼───────┐
                    │  PASS/WARN/  │◀────│  BFS traverse  │
                    │  BLOCK       │     │  reverse deps  │
                    │  + risk 0-1  │     │  + risk score  │
                    └──────────────┘     └───────────────┘
```

1. **Парсинг:** ts-morph сканирует проект на ESM-импорты, реэкспорты и CommonJS require
2. **Граф:** Строит граф зависимостей в памяти (без базы данных, без персистенции)
3. **Анализ:** BFS-обход обратных зависимостей от изменённых файлов
4. **Скоринг:** Риск = затронутые файлы / всего файлов (0-1)
5. **Вердикт:** PASS (< 60% порога), WARN (60-100%), BLOCK (> порога)

Поддерживает: ESM-импорты, ESM-реэкспорты, CommonJS `require()`, разрешение `.js` → `.ts` в стиле NodeNext.

| Лицензия | **MIT** | MIT | **BSL 1.1** | MIT |
| Количество инструментов | 4 | 30+ | 10 | 3 |
| Поддержка языков | TS/JS | 11 языков | Мульти | Мульти |
| Обнаружение циклов | **Да** | Да | Да | Нет |
| Оптимизация под агентов | **Да** | Частично | Частично | Частично |
| Локально / без облака | **Да** | Да | Да | Да |

**Когда выбрать CodeImpact MCP:** вам нужен быстрый, ограниченный ответ (PASS/WARN/BLOCK) перед коммитом — а не полноценный инструмент исследования кодовой базы. Нулевая настройка, MIT-лицензия, работает за секунды.

**Когда выбрать Codegraph/Depwire:** вам нужно глубокое исследование кодовой базы на множестве языков с персистентным хранилищем и визуализацией.

## FAQ

**Q: Обращается ли к сети?**
A: Нет. CodeImpact MCP на 100% локален. Он читает файлы проекта через ts-morph и никогда не делает сетевых запросов. Без API-ключей, без облака, без телеметрии.

**Q: Изменяет ли код?**
A: Нет. Все 4 инструмента — только для чтения (аннотированы `readOnlyHint: true`). Анализируют, но не пишут.

**Q: Насколько точен риск-скор?**
A: Риск-скор — это эвристика на основе графа (затронутые файлы / всего файлов). Он не учитывает поведение в рантайме, тесты или миграции данных. Воспринимайте его как сигнал для сортировки, а не как гарантию.

**Q: Работает ли с чисто JavaScript-проектами?**
A: Да. Поддерживаются файлы TypeScript и JavaScript (`.ts`, `.tsx`, `.js`, `.jsx`, `.mts`, `.cts`, `.mjs`, `.cjs`).

**Q: Какова скорость?**
A: Построение графа обычно занимает 1-5 секунд в зависимости от размера проекта. Вызовы инструментов на кешированном графе — практически мгновенные.

**Q: Кешируется ли граф?**
A: Да, граф кешируется в памяти по паре (projectRoot, tsconfigPath). Используйте `refresh_graph` для пересборки после значительных изменений.

## Ограничения

- Только TypeScript/JavaScript (мультиязычность не поддерживается)
- Не различает рантайм-импорты и импорты только типов
- Граф хранится только в памяти (не сохраняется между перезапусками сервера)
- Риск-скор структурный, а не семантический — не знает, какие файлы «важнее»
- Нет визуализации (только текст/JSON)
